# VibeKid 产品设计文档

> 让孩子通过和 AI 对话，把想法变成可以玩的作品。

## 1. 产品哲学

### 第一性原理

孩子的本质需求不是"学编程"，而是**把脑海中的想法变成现实**。传统编程教育（Scratch、Python 课）的瓶颈是：孩子必须先学会"说机器的语言"，才能创造。VibeKid 颠覆这个范式——AI 理解人的语言，孩子只需要清晰地表达想法。

### 核心原则

| 原则 | 含义 |
|---|---|
| **创作优先** | 孩子 focus 在"做什么"，而非"怎么做" |
| **增量可见** | 每一步都产出可运行版本，孩子随时能看到、能玩 |
| **主权在孩子** | AI 是助手，不是老师。所有产品决策（颜色、规则、玩法）由孩子做 |
| **零技术暴露** | 孩子不看代码、不知道编译、不接触任何技术概念 |
| **正向反馈循环** | 每个里程碑都有庆祝，让孩子感受到"我做到了" |

### 教育价值（对家长的叙事）

VibeKid 不教孩子写代码，但教会孩子更重要的东西：
- **需求表达**：把模糊的想法变成清晰的描述
- **项目拆解**：理解大目标需要分步实现
- **迭代思维**：产品从粗糙到精致的过程
- **与 AI 协作**：未来最核心的生产力技能

## 2. 用户旅程

### 2.1 总览

```
[进入平台] → [说出想法] → [AI 制定计划] → [逐步构建] → [优化打磨] → [分享作品]
```

### 2.2 详细流程（以"贪吃蛇"为例）

#### Phase 0: 进入

孩子打开 VibeKid，看到一个友好的界面：
- 上方是其他孩子的作品画廊（灵感来源）
- 中间是一个大大的输入框："你今天想做什么？"
- 下方是热门创意提示卡片："做一个小游戏"、"画一幅会动的画"、"做一个音乐盒"

#### Phase 1: 说出想法

```
孩子: 我想做一个贪吃蛇游戏！
```

#### Phase 2: AI 制定计划

AI（角色名：小V）回复：

```
小V: 贪吃蛇！太棒了！🎮 我们一起来做！

我想了一下，我们可以分这几步来完成：

📋 我们的计划：
 ☐ 第1步：让一条小蛇出现在屏幕上
 ☐ 第2步：让小蛇可以自己移动
 ☐ 第3步：用键盘控制小蛇的方向
 ☐ 第4步：放一些食物让小蛇吃
 ☐ 第5步：吃到食物小蛇会变长，还能得分
 ☐ 第6步：撞到自己或墙壁游戏结束

准备好了吗？我们先从第1步开始！
```

**关键设计**：计划始终显示在界面侧边，当前步骤高亮，已完成步骤打勾。孩子随时能看到进度。

#### Phase 3: 逐步构建（核心循环）

每一步遵循相同模式：

```
[AI 实现] → [孩子预览/试玩] → [AI 问业务问题] → [孩子调整] → [确认进入下一步]
```

**第1步示例**：

```
小V: 好啦！你的小蛇出现了！看看右边的预览 →

（预览区域显示一个绿色方块在灰色背景上）

小V: 你觉得小蛇应该是什么颜色的？
孩子: 我想要蓝色的！
小V: 好的，蓝色的小蛇！（预览更新）背景呢？你喜欢什么颜色？
孩子: 黑色！像太空一样！
小V: 酷！太空蛇！🚀（预览更新为黑色背景 + 蓝色蛇）

满意了吗？我们进入第2步，让小蛇动起来！
```

**第3步示例**（展示如何处理不同设备）：

```
小V: 现在你可以用键盘的方向键控制小蛇了！试试！

（孩子试玩）

小V: 感觉怎么样？小蛇移动的速度合适吗？
孩子: 太慢了！
小V: 好，加速！（更新）再试试？
孩子: 可以再快一点吗
小V: 没问题！（更新）
孩子: 嗯这个速度刚好！
小V: 太好了！✅ 第3步完成！我们已经完成一半了！
```

#### Phase 4: 优化打磨

基础步骤完成后，孩子可以继续提需求：

```
孩子: 我想加一个开始界面
孩子: 吃到食物的时候要有音效
孩子: 我想加一个最高分记录
```

AI 会继续实现，每次都是"实现 → 预览 → 调整"循环。

#### Phase 5: 分享

```
小V: 你的贪吃蛇游戏做好了！🎉
     想给朋友看看吗？点击"分享"按钮，就能生成一个链接！

（孩子点击分享，获得一个链接，朋友打开就能玩）
```

### 3.2 浏览器端运行时

#### 为什么是 HTML/CSS/JS + iframe

| 维度 | 说明 |
|---|---|
| 零编译 | JS 是浏览器原生语言，无需编译步骤 |
| 即时反馈 | 代码注入后立即运行，孩子秒看结果 |
| 安全隔离 | `<iframe sandbox>` 提供完善的沙箱 |
| 丰富生态 | p5.js、Tone.js、matter.js 等库覆盖大部分创作场景 |
| 易于分享 | 最终产物就是一个 HTML 文件，可直接部署 |

#### 预装库

| 库 | 用途 | 典型场景 |
|---|---|---|
| **p5.js** | 创意编程/图形/动画 | 游戏、动画、绘画 |
| **Tone.js** | 声音/音乐 | 音乐盒、音效 |
| **matter.js** | 2D 物理引擎 | 弹球、物理模拟 |
| **anime.js** | 动画 | UI 动效、故事动画 |

#### iframe 沙箱策略

```html
<iframe
  sandbox="allow-scripts"
  <!-- 不允许: allow-same-origin, allow-forms, allow-popups -->
  <!-- 完全隔离，只允许脚本执行 -->
></iframe>
```

代码通过 `srcdoc` 或 `Blob URL` 注入，与主页面完全隔离。

#### 主页面与 iframe 通信

使用 `postMessage` 实现双向通信：
- 主页面 → iframe：注入新代码、重置状态
- iframe → 主页面：报告运行结果、错误信息（用于 AI 自动修复）

### 3.3 服务端 AI Engine

#### 核心职责

1. **理解孩子意图**：将自然语言转化为产品需求
2. **制定计划**：将大目标拆解为可执行步骤
3. **生成代码**：根据当前步骤生成完整可运行代码
4. **增量更新**：在已有代码基础上修改，而不是每次重写
5. **错误自愈**：如果运行报错，AI 自动修复，孩子无感知

#### System Prompt 设计核心

```
你是小V，一个帮助孩子创作的 AI 助手。

核心规则：
1. 你的回复中永远不出现任何代码、技术术语
2. 你只问孩子关于产品的问题（颜色、速度、规则等）
3. 每次对话都要产出可运行的版本
4. 用孩子能理解的语言，简短、有趣、充满鼓励
5. 每完成一个里程碑，要庆祝

代码生成规则：
1. 生成完整的 HTML 文件（内含 CSS 和 JS）
2. 使用预装库（p5.js 等）
3. 代码必须在 iframe sandbox 中可运行
4. 每次生成的代码都是完整可运行版本
5. 保持代码简洁，优先使用简单实现
```

### 3.4 数据模型

#### Project（项目）

```
Project {
  id: string
  userId: string
  title: string              // "我的贪吃蛇"
  description: string        // AI 生成的描述
  plan: PlanStep[]           // 计划步骤
  currentStep: number        // 当前进行到第几步
  state: "planning" | "building" | "polishing" | "done"
  versions: Version[]        // 所有历史版本
  currentVersion: number     // 当前版本号
  conversation: Message[]    // 完整对话记录
  sharedUrl: string?         // 分享链接
  createdAt: datetime
  updatedAt: datetime
}
```

#### PlanStep（计划步骤）

```
PlanStep {
  index: number
  title: string              // "让小蛇出现在屏幕上"
  status: "pending" | "active" | "done"
  versionId: string?         // 完成时对应的版本
}
```

#### Version（版本）

```
Version {
  id: string
  number: number
  code: string               // 完整 HTML/CSS/JS 代码（孩子不可见）
  description: string        // "加了蓝色小蛇和黑色背景"
  createdAt: datetime
}
```

#### Message（对话消息）

```
Message {
  id: string
  role: "kid" | "assistant"
  content: string            // 对话内容（不含代码）
  codeGenerated: boolean     // 这条消息是否触发了代码更新
  versionId: string?         // 关联的版本
  timestamp: datetime
}
```

## 5. 技术选型

### 5.1 前端

| 技术 | 选型 | 理由 |
|---|---|---|
| 框架 | **Next.js (React)** | SSR 优化首屏、App Router、成熟生态 |
| 语言 | **TypeScript** | 类型安全，开发体验好 |
| 样式 | **Tailwind CSS** | 快速开发，响应式友好 |
| 状态管理 | **Zustand** | 轻量、简单 |
| 实时通信 | **SSE (Server-Sent Events)** | AI 流式输出，比 WebSocket 更简单 |
| 编辑器预览 | **sandboxed iframe** | 安全隔离执行用户代码 |

### 5.2 后端

| 技术 | 选型 | 理由 |
|---|---|---|
| 运行时 | **Node.js** | 与前端统一技术栈 |
| 框架 | **Next.js API Routes** | 全栈统一，部署简单 |
| AI | **OpenAI / Anthropic API** | 主流 LLM，代码生成能力强 |
| 数据库 | **PostgreSQL + Prisma** | 可靠、结构化、ORM 友好 |
| 认证 | **NextAuth.js** | 支持多种登录方式 |
| 存储 | **S3 兼容** | 存储分享的静态文件 |
| 部署 | **Vercel / AWS** | Next.js 原生支持 |

### 5.3 AI Prompt 架构

采用**双层 Prompt 分离**：

```
┌─────────────────────────────┐
│  面向孩子的对话层             │  ← 生成友好的对话内容
│  (不含任何代码)              │
├─────────────────────────────┤
│  面向代码的生成层             │  ← 生成 HTML/CSS/JS 代码
│  (结构化 JSON 输出)          │
└─────────────────────────────┘
```

一次 AI 调用同时产出两部分（通过 structured output）：
1. **对话内容**：给孩子看的友好文字
2. **代码更新**：注入 iframe 的完整代码（孩子不可见）

这样可以用一次 API 调用完成两件事，避免不一致。

## 6. 安全设计

### 6.1 iframe 沙箱

```
sandbox="allow-scripts"
```

**不允许**:
- `allow-same-origin`：防止访问主页面数据
- `allow-forms`：防止表单提交
- `allow-popups`：防止弹窗
- `allow-top-navigation`：防止跳转

### 6.2 内容安全

- AI 生成的代码会经过安全审查（禁止 `fetch`、`XMLHttpRequest`、`eval` 等）
- CSP 头限制 iframe 中的网络访问
- 定期扫描生成的代码，防止注入

### 6.3 用户安全（针对儿童）

- **COPPA 合规**：不收集 13 岁以下儿童个人信息（或需家长同意）
- **内容过滤**：AI 输入/输出都经过内容安全过滤
- **分享审核**：公开分享的作品需经过自动化内容审核
- **家长控制**：家长可查看孩子的所有对话和作品

## 10. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| AI 生成的代码有 bug | 孩子看到空白/报错 | 错误自愈机制：iframe 报错 → AI 自动修复 → 重新渲染 |
